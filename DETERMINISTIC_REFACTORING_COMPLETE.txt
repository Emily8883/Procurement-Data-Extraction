2. PROCUREMENT PIPELINE (procurement_pipeline.py)
───────────────────────────────────────────────────
Specialization for invoices/purchase orders:

Extract:
  - Supplier, invoice/PO number, date, currency
  - Financial fields (subtotal, tax, total)
  - Line items (description, qty, unit price, total)
  - Metadata (confidence, missing fields, method)

Validate (STRICT):
  - All critical fields required (no exceptions)
  - Math check: subtotal + tax ≈ total_amount
  - Line item check: qty × unit_price ≈ total_price
  - Metadata detection: quantity > 10,000 = suspicious
  - Confidence minimum: 30%
  - Errors block processing
  - Warnings block processing (STRICT mode)

Finalize:
  - Output schema optimized for invoices
  - Include all fields
  - Mark as "invoice_or_po"
  - Include line item count

3. SPECIFICATION PIPELINE (specification_pipeline.py)
───────────────────────────────────────────────────────
Specialization for specs/datasheets/requirements:

Extract:
  - Document name, line items (technical content)
  - Optional: supplier, date, currency (warnings if missing)
  - Technical specifications detection
  - No pricing extraction (not in specs)

Validate (LENIENT):
  - Only critical: document name + line items
  - Optional fields = warnings (not errors)
  - NO math validation (specs have no prices)
  - Higher metadata threshold: qty > 100,000
  - Confidence minimum: 20%
  - Errors block processing
  - Warnings DO NOT block (LENIENT mode)

Finalize:
  - Output schema optimized for specifications
  - Exclude pricing fields (intentional)
  - Include technical specifications
  - Mark as "specification"

4. DOCUMENT ROUTER (document_router.py)
───────────────────────────────────────
Deterministic classification to routing:

Algorithm:
  1. Extract text from PDF (first 3 pages)
  2. Score procurement likelihood
     - Keyword matches (supplier, invoice, etc.)
     - Field presence (supplier, pricing, date)
     - Text patterns (total, subtotal, precio)
  3. Score specification likelihood
     - Keyword matches (specification, requirement, etc.)
     - Field absence (no supplier, no pricing)
     - Technical patterns (standard, norm, temperature)
  4. Compare scores
     - If procurement > specification → PROCUREMENT
     - If specification > procurement → SPECIFICATION
     - If tied → Apply fallback rules

Fallback Rules (deterministic tie-breaking):
  1. Has explicit supplier → PROCUREMENT
  2. Has explicit pricing → PROCUREMENT
  3. Has explicit technical requirements → SPECIFICATION
  4. No pricing data → SPECIFICATION
  5. Default → PROCUREMENT (configurable)

Result: (pipeline_type, confidence_0_to_1)
  - "procurement", 0.87
  - "specification", 0.72
