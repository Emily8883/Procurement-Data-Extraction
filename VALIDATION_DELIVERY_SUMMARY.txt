VALIDATION RULES IMPLEMENTED
================================================================================

RULE 1: FIELD COMPLETENESS CHECK ✅
────────────────────────────────────
Purpose: Ensure critical fields are present

Critical Fields:
  • supplier
  • invoice_or_po_number
  • issue_date
  • total_amount

Validation Modes:
  • TEXT-BASED: STRICT (all fields required)
  • SCANNED: LENIENT (fields expected but not required due to OCR)

Output: field_checks dictionary with presence status per field


RULE 2: MATH CONSISTENCY CHECK ✅
──────────────────────────────────
Purpose: Verify financial calculations are consistent

Formula: subtotal + tax ≈ total_amount
Tolerance: ±5% (configurable)

Process:
  1. Parse numeric values from fields
  2. Calculate: expected = subtotal + tax
  3. Compare with total_amount
  4. Calculate percentage difference
  5. Flag if exceeds tolerance

Output: math_checks dictionary
  • subtotal_match: true/false
  • line_item_accuracy: true/false


RULE 3: LINE ITEM VALIDATION ✅
────────────────────────────────
Purpose: Verify line-level pricing accuracy

Formula: quantity × unit_price ≈ total_price (per line)
Tolerance: ±5% per line item
Metadata Detection: Flags quantities > 10,000

Process:
  1. For each line item extract: qty, unit_price, total_price
  2. Calculate: expected_total = qty × unit_price
  3. Compare with line total_price
  4. Flag if exceeds tolerance OR qty seems suspicious
  5. Flag if components missing

Output: line_item_checks array
  • Per-item validation status
  • Specific issues identified
  • Suspicious quantity detection


RULE 4: OCR QUALITY CHECK ✅
─────────────────────────────
Purpose: Assess quality of OCR extraction (scanned docs)

Checks:
  1. Confidence Score Assessment
     └─ Very low: flags if 0% (no extraction)
     └─ Low: flags if < 20%

  2. Garbled Text Detection
     └─ Scans for non-standard Unicode patterns
     └─ Identifies potential OCR artifacts

  3. Missing Rows Indication
     └─ Flags if extremely few items (1) when many expected

Output: ocr_checks dictionary
  • has_garbled_text: true/false
  • has_missing_rows: true/false
  • confidence_very_low: true/false
  • issues: [list of specific OCR problems]

not traditional invoices or purchase orders.

Evidence:
  • Complete absence of pricing (intentional - not yet bid)
  • No issue/approval dates (not finalized)
  • Technical specification content (requirements, standards)
  • Line items include regulation references and standards

Implication:
  • Extraction rules are working correctly (detecting what's there)
  • Validation "failures" are proper detection of expected missing data
  • Documents are properly classified but have different field requirements
  • This is NOT an error - it's correct behavior

Resolution Options:
  A) Modify validation rules to make price/date optional for specifications
  B) Separate pipeline for specifications with different rules
  C) Update document classification to distinguish invoice vs specification
  D) Manual approval workflow for specification documents


FINDING #2: Metadata Captured as Line Items ⚠️
─────────────────────────────────────────────
Some line items contain regulation references that look like quantities:

Examples from 0c2f9fc842a866ee.pdf:
  • "Ordenanza Nº 2537" → extracted as quantity: 2537
  • "Artículo 93" → extracted as quantity: 93
  • "Decreto 29783" → extracted as quantity: 29783

Root Cause:
  • Regex patterns designed for invoices too aggressive for specs
  • No context awareness to distinguish metadata from line items
  • Spanish specifications use different formatting

Mitigation Applied:
  ✓ Validator flags quantities > 10,000 as suspicious
  ✓ Reports in line_item_checks with "suspicious quantity" warning
  ✓ Does NOT delete - flags for review

Improvement Needed:
  • Refine patterns for Spanish specifications
  • Add context detection for regulation numbers
  • Consider manual inspection for specification documents
