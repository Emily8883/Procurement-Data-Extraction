VALIDATION ARCHITECTURE
================================================================================

The validation layer performs STRICT financial consistency checks with NO 
auto-correction. Issues are detected and flagged for manual review.

VALIDATION RULES IMPLEMENTED:

1. ✅ FIELD COMPLETENESS CHECK
   ├─ Critical fields: supplier, invoice_or_po_number, issue_date, total_amount
   ├─ Scope: TEXT-based documents (STRICT)
   ├─ Scope: SCANNED documents (LENIENT - due to OCR limitations)
   └─ Detection: Missing field tracking with field_checks dict

2. ✅ MATH CONSISTENCY CHECK  
   ├─ Rule: subtotal + tax ≈ total_amount
   ├─ Tolerance: ±5% (configurable)
   ├─ Validation: Percentage difference calculation with range validation
   └─ Output: math_checks dict (subtotal_match: true/false)

3. ✅ LINE ITEM VALIDATION
   ├─ Rule: quantity × unit_price ≈ total_price
   ├─ Tolerance: ±5% per line item
   ├─ Metadata Detection: Flags quantities > 10,000 (likely regulation numbers)
   ├─ Incomplete Data: Flags missing qty/price components
   └─ Output: line_item_checks array with per-item validation

4. ✅ OCR QUALITY CHECK
   ├─ Confidence Score: Flags if 0% (no extraction)
   ├─ Very Low Confidence: Flags if < 20%
   ├─ Garbled Text: Detects non-standard Unicode patterns
   ├─ Missing Rows: Flags if suspicious (1 item when many expected)
   └─ Output: ocr_checks dict with issue tracking

5. ✅ ENHANCED OUTPUT STRUCTURE
   ├─ All original fields preserved
   ├─ New "validation" key added to each record
   └─ Structure includes: is_valid, errors, warnings, checks

FINDING 1: DOCUMENT TYPE MISMATCH

The 2 invalid text-based documents are PROCUREMENT SPECIFICATIONS,
not invoices or purchase orders.

Evidence:
  • Complete lack of pricing information
  • No issue dates or formal approval dates
  • Technical specification content (requirements, standards)
  • Line items contain regulation numbers and metadata

Implication:
  • Extraction rules designed for invoices don't apply here
  • Validation "failures" are actually correct behavior
  • Documents lack pricing/date data BY DESIGN

Resolution Options:
  A) Update validation rules to accept specifications (set prices/dates as optional)
  B) Separate extraction pipelines for invoices vs specifications
  C) Update document classification to distinguish invoice vs spec

FINDING 2: METADATA CAPTURED AS LINE ITEMS
────────────────────────────────────────────
Line item extraction captured regulation numbers as quantities:

  • "Artículo 93" → quantity: 93
  • "Ordenanza Nº 2537" → quantity: 2537
  • "Decreto 29783" → quantity: 29783

Root Cause:
  • Regex patterns too aggressive in specification documents
  • No context awareness to distinguish metadata from items

Mitigation:
  • Validator flags quantities > 10,000 as suspicious ✓
  • Pattern refinement needed for Spanish specifications
  • Consider manual inspection for specification documents
